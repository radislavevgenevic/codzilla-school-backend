2. ПРАКТИЧЕСКАЯ ЧАСТЬ
2.1. Проектирование и разработка программного средства

2.1.1. Карта сайта

Проектирование информационной архитектуры системы началось с разработки карты сайта, которая определяет иерархическую структуру страниц и функциональных разделов веб-приложения. Карта сайта отражает разделение прав доступа для различных ролей пользователей: администраторов, преподавателей, родителей и неавторизованных посетителей.

Публичная часть (для всех пользователей):
— Главная страница (преимущества, общая информация).
— Каталог курсов (список доступных направлений обучения).
— Страница курса (детальное описание, учебный план).
— Форма подачи заявки на обучение.
— Раздел статистики (показатели работы школы).
— Форма обратной связи.

Авторизация и профиль:
— Страница входа в систему.
— Регистрация (преимущественно для родителей).
— Личный кабинет пользователя (управление персональными данными).

Панель администратора (Admin Panel):
— Управление курсами (создание, редактирование программ).
— Управление группами (формирование составов, назначение расписания).
— Реестр пользователей (администрирование учетных записей).
— Модуль управления преподавателями (назначение на курсы).
— База учеников (карточки обучающихся).
— Управление подписками и финансовая отчетность.
— Модуль посещаемости и уведомлений.

Интерфейс преподавателя:
— Мои курсы и группы.
— Журнал посещаемости.
— Расписание занятий.
— Данные о заработной плате.

Кабинет родителя:
— Список детей.
— Информация об успеваемости и посещаемости.
— Управление активными подписками.

2.1.2. Этап эскизного проектирования

На этапе эскизного проектирования были определены основные визуальные концепции и принципы взаимодействия пользователя с интерфейсом. Основной упор был сделан на чистоту интерфейса, интуитивность навигации и адаптивность.

Для разработки эскизов использовался компонентный подход. Выделены ключевые элементы интерфейса:
— Навигационные панели (Sidebar для внутренних разделов и Navbar для публичной части).
— Информационные карточки (для отображения курсов, групп и статистики).
— Табличные представления для управления данными в админ-панели.
— Модальные окна для форм ввода и подтверждения действий.

Особое внимание уделено адаптивности (Mobile-first подход). В коде фронтенд-части реализованы специализированные хуки (например, useIsMobile), обеспечивающие корректное отображение сложных интерфейсных решений (таких как журналы посещаемости или графики расписания) на мобильных устройствах и планшетах.

2.1.3. Архитектура веб-приложения

Архитектура разрабатываемой системы построена на разделении клиентской и серверной частей (decoupling), что обеспечивает масштабируемость и независимость разработки.

Серверная часть (Backend) реализована на фреймворке Laravel 11. Она функционирует как RESTful API, предоставляя данные в формате JSON. Для обеспечения безопасности и авторизации используется Laravel Sanctum, реализующий механизм аутентификации на основе токенов (Stateful/Token-based auth).

Клиентская часть (Frontend) построена на базе Next.js 15 с использованием App Router. Архитектура фронтенда следует методологии Feature-Sliced Design (FSD), что позволяет эффективно структурировать код по слоям (app, pages, widgets, features, entities, shared). Это обеспечивает высокую переиспользуемость компонентов и легкость поддержки кода.

Взаимодействие между клиентом и сервером осуществляется посредством HTTP-запросов через библиотеку Axios. Архитектура является трехзвенной:
1. Слой представления (Next.js components).
2. Слой бизнес-логики (Laravel controllers и services).
3. Слой данных (PostgreSQL/MySQL база данных, управляемая через Eloquent ORM).

2.1.4. Проектирование структуры базы данных

Логическая структура базы данных спроектирована с учетом обеспечения целостности данных и минимизации избыточности. Основные сущности системы и их связи:

1. Пользователи (users): хранит данные об учетных записях всех ролей (admin, teacher, parent).
2. Курсы (courses): содержит информацию об учебных программах, их описании и стоимости.
3. Ученики (students): профили детей, связанные с родителем (parent_id) связью «один ко многим».
4. Группы (groups): объединяют учеников для проведения занятий. Связь с учениками реализована через промежуточную таблицу (group_student).
5. Расписание (schedules): определяет время проведения занятий для конкретных групп.
6. Занятия (lessons): конкретные экземпляры уроков в рамках расписания, содержат данные о домашних заданиях.
7. Посещаемость (attendance): фиксирует присутствие ученика на конкретном занятии и статус (присутствовал, пропустил по уважительной причине и т.д.).
8. Подписки (subscriptions): хранят информацию об оплаченных периодах обучения, связанных с учеником и курсом.
9. Заявки (applications): содержат данные от потенциальных клиентов, поступившие через публичную форму.

Связи между таблицами реализованы через внешние ключи с каскадным обновлением и ограничением удаления для критически важных данных, что гарантирует ссылочную целостность системы.

2.1.5. Проектирование серверной части (Laravel)

Проектирование серверной части системы базируется на архитектурном паттерне MVC (Model-View-Controller), адаптированном под нужды API. В качестве основы выбран фреймворк Laravel за его мощные инструменты работы с базой данных, встроенную систему миграций и развитые механизмы безопасности.

Ключевые аспекты серверной реализации:
— Маршрутизация (Routing): Все API-эндпоинты сгруппированы по версиям (v1) и функциональным зонам (public, admin, parent, teacher). Это позволяет четко разграничить доступ на уровне промежуточного ПО (Middleware).
— Контроллеры (Controllers): Логика обработки запросов вынесена в специализированные контроллеры. Например, AuthController отвечает за жизненный цикл сессии пользователя, а AttendanceController в пространстве имен Admin управляет процессами учета посещаемости.
— Ресурсы (API Resources): Для преобразования моделей Eloquent в стандартизированные JSON-ответы используются классы-ресурсы (например, CourseResource, StudentResource). Это позволяет абстрагировать структуру БД от внешнего интерфейса.
— Валидация: Каждый входящий запрос проходит через строгую систему правил валидации, реализованную непосредственно в методах контроллеров или специализированных классах Request.
— Сервисный слой (Services): Сложная бизнес-логика, не относящаяся напрямую к обработке HTTP-запросов, вынесена в сервисы (например, CourseApplicationNotifier), что соответствует принципу единственной ответственности (SRP).

2.1.6. Проектирование клиентской части (Next.js)

Клиентская часть спроектирована как современное одностраничное приложение (SPA) с элементами серверного рендеринга (SSR), предоставляемыми фреймворком Next.js. Использование методологии Feature-Sliced Design (FSD) позволило создать масштабируемую структуру проекта.

Основные проектные решения фронтенда:
— Управление состоянием: Для хранения данных пользователя и управления состоянием аутентификации используется React Context API (AuthProvider).
— Пользовательские хуки (Hooks): Логика взаимодействия с API инкапсулирована в кастомные хуки (useAuth, useCourses, useProfileSummary и др.), что упрощает код компонентов и делает его более читаемым.
— Модульная структура: Каждая функциональная область (Auth, Course, Profile) содержит свои собственные компоненты, API-сервисы и модели данных. Например, в фиче Profile выделены отдельные менеджеры для управления курсами, группами и учениками (useAdminCoursesManager, useAdminGroupsManager).
— Динамическая маршрутизация: Использование файловой системы App Router в Next.js позволяет эффективно управлять вложенными интерфейсами, особенно в сложных панелях управления (dashboard), где требуется отображение различных подразделов в рамках единого макета (layout).
— Интерфейсные компоненты: Библиотека компонентов разделена на переиспользуемые атомарные элементы (UI-кит в слое shared) и более сложные виджеты.

2.1.7. Реализация системы авторизации

Система авторизации построена на базе Laravel Sanctum и обеспечивает безопасный доступ к функционалу приложения в соответствии с ролью пользователя. При входе в систему сервер генерирует уникальный токен доступа (Bearer Token), который сохраняется на стороне клиента (в cookies или localStorage) и используется для аутентификации последующих запросов.

Реализованы следующие механизмы:
— Регистрация: Новые пользователи (преимущественно родители) могут создать учетную запись, заполнив форму. По умолчанию им назначается роль 'parent'.
— Аутентификация: Проверка email и пароля с использованием механизма хеширования Bcrypt.
— Контроль доступа (RBAC): С помощью Middleware в Laravel реализована проверка прав доступа. Пользователь с ролью 'admin' имеет полный доступ, 'teacher' — к учебным журналам и расписанию, 'parent' — только к данным своих детей.
— Восстановление сессии: При обновлении страницы фронтенд выполняет запрос к эндпоинту /auth/me для получения актуальных данных о пользователе и его правах.

2.1.8. Реализация функционала управления курсами

Модуль управления курсами является центральным элементом системы. Реализация включает:
— CRUD-интерфейс для администратора: возможность создания новых направлений обучения, загрузки иконок и описаний.
— Публичный каталог: фильтрация и отображение доступных курсов для неавторизованных пользователей.
— Интеграция с заявками: при просмотре курса пользователь может оставить заявку, которая автоматически привязывается к выбранной программе обучения в базе данных.

2.1.9. Реализация функционала управления учениками

Система обеспечивает ведение полноценного реестра обучающихся. В административной панели реализована картотека учеников, где хранится информация о персональных данных, привязке к родителю и текущем статусе обучения.
Функционал включает:
— Добавление и редактирование профилей учеников.
— Управление составом учебных групп (добавление/исключение учеников из групп).
— Отслеживание истории обучения каждого ребенка.

2.1.10. Реализация учета посещаемости

Модуль посещаемости автоматизирует процесс ведения классных журналов. Преподаватели и администраторы имеют доступ к интерфейсу отметки присутствующих.
Особенности реализации:
— Генерация списка студентов на основе выбранного занятия в расписании.
— Выбор статуса присутствия (присутствовал, пропустил, пропустил по уважительной причине).
— Автоматическое сохранение данных в таблицу attendance с привязкой к уроку и студенту.
— Отображение истории посещаемости для родителей в их личном кабинете.

2.1.11. Реализация личного кабинета родителя

Для родителей разработан специализированный интерфейс, решающий задачу прозрачности учебного процесса.
Основные функции кабинета:
— Отображение списка всех детей пользователя, обучающихся в школе.
— Просмотр детализированной статистики по каждому ребенку: посещаемость уроков, активные курсы, текущий прогресс.
— Информация о подписках: статус оплаты, срок действия абонемента на обучение.

2.1.12. Разработка пользовательского интерфейса

Пользовательский интерфейс (UI) разработан с использованием CSS-фреймворка Tailwind CSS и библиотеки компонентов. Дизайн выполнен в современной минималистичной стилистике.
Основные интерфейсные решения:
— Использование боковых панелей навигации (Sidebar) для эффективного переключения между функциональными модулями в административной и преподавательской частях.
— Информативные дашборды с ключевыми показателями.
— Интерактивные таблицы с возможностью фильтрации и поиска.
— Адаптивная верстка, обеспечивающая корректную работу системы на смартфонах и планшетах.

2.1.13. Разработка алгоритмов работы системы

В системе реализован ряд алгоритмов, обеспечивающих автоматизацию бизнес-процессов:
— Алгоритм расчета заработной платы преподавателя: на основе количества проведенных занятий за определенный период (день, неделя, месяц).
— Алгоритм фильтрации курсов: динамическая обработка параметров запроса на фронтенде и бэкенде для быстрого поиска подходящих программ.
— Алгоритм уведомлений: автоматическое оповещение администраторов о новых заявках через сервис CourseApplicationNotifier.
— Алгоритм управления расписанием: логика проверки пересечений и корректного отображения занятий в календаре.

2.2. Тестирование программного средства

2.2.1. Функциональное тестирование

Целью функционального тестирования является проверка соответствия разработанной системы заявленным требованиям. Тестирование проводилось как в автоматизированном режиме (с использованием PHPUnit для бэкенда), так и вручную по заранее подготовленным сценариям.
В ходе тестирования проверялись все ключевые бизнес-сценарии: от подачи заявки на сайте до формирования отчетов администратором.

2.2.2. Тестирование системы авторизации

Тестирование модуля безопасности включало проверку:
— Корректности регистрации новых пользователей с валидацией уникальности email и надежности пароля.
— Процесса входа в систему и генерации токенов Sanctum.
— Работы Middleware: попытки доступа к административным эндпоинтам под учетной записью родителя блокировались с кодом ответа 403 Forbidden.
— Механизма выхода из системы (Logout) и деактивации токена.

2.2.3. Тестирование модуля управления курсами

В рамках тестирования данного модуля проверялись:
— Создание, редактирование и удаление курсов в административной панели.
— Отображение изменений в публичном каталоге в режиме реального времени.
— Загрузка медиа-файлов (иконок курсов) и корректность их путей в API.
— Работа поисковых фильтров по различным критериям сложности и направлениям.

2.2.4. Тестирование модуля учета посещаемости

Тестирование журнала посещаемости фокусировалось на:
— Корректности выборки студентов для конкретной группы и занятия.
— Надежности сохранения отметок о присутствии при множественном выборе.
— Целостности данных: проверка того, что отметка посещаемости не может быть поставлена на несуществующий урок.
— Синхронизации: мгновенное отображение отметок в личном кабинете родителя после их сохранения преподавателем.

2.2.5. Анализ выявленных ошибок и их устранение

В процессе разработки и тестирования был выявлен ряд дефектов, которые были оперативно устранены:
— Проблема: Некорректное отображение времени занятий из-за разницы в часовых поясах сервера и клиента. Решение: Переход на хранение времени в формате UTC и конвертацию в локальное время пользователя на фронтенде.
— Проблема: Дублирование заявок при быстром многократном нажатии на кнопку «Отправить». Решение: Реализована блокировка кнопки на стороне клиента и проверка уникальности активных заявок на стороне сервера.
— Проблема: Ошибка при удалении курса, к которому привязаны активные группы. Решение: Внедрена проверка зависимостей и «мягкое удаление» (Soft Deletes) для сохранения истории.

2.3. Документальное описание программного продукта

2.3.1. Назначение и условия применения

Разработанная информационная система предназначена для автоматизации операционной деятельности частных школ программирования. Система позволяет управлять учебным процессом, контролировать посещаемость, вести финансовый учет подписок и обеспечивать взаимодействие с родителями учеников.
Условием применения является наличие веб-сервера для хостинга бэкенда и фронтенд-части, а также доступ к сети Интернет для пользователей.

2.3.2. Требования к аппаратному и программному обеспечению

Для работы системы на стороне сервера:
— Процессор: от 2 ядер.
— Оперативная память: от 4 ГБ RAM.
— Программное обеспечение: PHP 8.3+, Node.js 20+, MySQL 8.0+ или PostgreSQL 15+.

Для работы на стороне пользователя:
— Любой современный веб-браузер (Chrome, Firefox, Safari, Edge) актуальной версии.
— Стабильное интернет-соединение.

2.3.3. Руководство по установке и развертыванию

Процесс развертывания включает следующие шаги:
1. Клонирование репозиториев бэкенда и фронтенда.
2. Настройка серверной части:
   — Установка зависимостей: composer install.
   — Настройка файла .env (подключение к БД, ключи Sanctum).
   — Выполнение миграций: php artisan migrate --seed.
3. Настройка клиентской части:
   — Установка зависимостей: npm install.
   — Настройка переменной NEXT_PUBLIC_API_URL в .env.local.
   — Сборка приложения: npm run build.
4. Настройка веб-сервера (Nginx/Apache) для проксирования запросов.

2.3.4. Руководство пользователя (администратор)

Администратор имеет полный доступ к системе. Для начала работы необходимо:
1. Авторизоваться в панели управления.
2. Создать курсы в разделе «Управление курсами».
3. Сформировать учебные группы и назначить им преподавателей.
4. В разделе «Заявки» обрабатывать входящие запросы от клиентов и зачислять учеников в группы.
5. Контролировать финансовые поступления в разделе «Подписки».

2.3.5. Руководство пользователя (родитель)

После регистрации и зачисления ребенка родителю доступен следующий функционал:
1. В разделе «Мои дети» просмотр списка обучающихся детей.
2. Клик на профиль ребенка открывает детальный журнал посещаемости и комментарии преподавателей.
3. Раздел «Оплата» позволяет отслеживать статус текущих абонементов и сроки их окончания.

2.3.6. Руководство пользователя (преподаватель)

Работа преподавателя в системе включает:
1. Просмотр расписания в разделе «Календарь».
2. Выбор текущего занятия для отметки посещаемости.
3. Заполнение данных о домашнем задании для каждой группы.
4. Просмотр детализации заработной платы на основе фактически проведенных уроков в разделе «Зарплата».
